# CallBrain — Stack Decision & Apple-Silicon Optimization (amends ARCHITECTURE.md)

> Synthesis of six bake-off lanes into one binding decision. **The big fork is resolved: option E (Swift-native-first). There is zero Python in the shipped `.app` and in the runtime hot path.** Every layer below is the #1 / best-in-class pick for *this* app on *this* hardware (Apple Silicon, macOS 26.5.1, Swift 6.3.2). LLM generation is a CLI subprocess (`claude -p` / `codex exec`) in every option, so it is not an architectural differentiator.
>
> **Honest framing (read §1A first).** "Local-first" here means **local index + local retrieval + on-device transcription/diarization/embeddings**, with **cloud generation by default**: Ask-AI ships retrieved transcript context to a third-party cloud LLM (Anthropic/OpenAI) on every answer. This is a deliberate, disclosed trade — not a "fully offline" app. The phrase "private, local-first" used elsewhere in this doc as design *justification* (e.g. rejecting `allow-unsigned-executable-memory`, favoring SQLCipher) is about the **at-rest index and the launch/hot path**, which are genuinely on-device; it is NOT a claim that meeting content never leaves the machine. See §1A.

## 1A. Data Egress & Privacy Threat Model (read before trusting "local-first")

CallBrain is **local index + cloud generation by default.** Being explicit about what leaves the device is mandatory for an app holding years of confidential crypto/decentralized-AI meeting content.

**What stays on device (genuinely local):**
- Raw audio/video, full transcripts, diarization, the SQLite source of truth, FTS5, embeddings (BLOBs), and the usearch/sqlite-vec index. At-rest encryption via the optional SQLCipher path.
- Transcription, diarization, embedding, retrieval, and ranking — all in-process on actors. No network needed for ingest/index/search (after first-run model download — see below and §6).

**What leaves the device (Ask-AI / RAG generation):**
- On every Ask-AI answer, the `LLMRunner` subprocess (`claude -p` / `codex exec`) sends a **prompt containing retrieved transcript chunks** plus the user's question to a **third-party cloud LLM (Anthropic/OpenAI)**. **Confidential meeting content egresses off-device on every cloud RAG answer.** SQLCipher-at-rest does nothing to protect this; it is an egress, not an at-rest, concern.

**Egress minimization (must be implemented, not assumed):**
1. **Chunk minimization** — send only the top-k retrieved chunks actually used (cap k and total tokens), never whole transcripts or whole meetings.
2. **Optional redaction pass** — before egress, optionally scrub configurable sensitive tokens (wallet addresses, private keys, counterparty names) via a local pass; surface what was redacted.
3. **Per-answer disclosure** — the UI must make it visible that an Ask-AI answer used a cloud model and which meetings' chunks were sent (provenance already hydrated by `chunk_id`).
4. **Provider choice is an egress choice** — `claude` vs `codex` selects which cloud vendor receives content; expose this, don't hide it behind a "model" toggle.

**True local-only mode (optional, shipped, VERIFY):** a user-selectable **"Private mode (no cloud)"** that routes generation to an on-device LLM (MLX-Swift / `llama.cpp` GGUF subprocess) so a zero-egress path exists for sensitive sessions. This is the honest answer to "an app holding years of private crypto calls." Default remains cloud (better quality/latency); the local path's model id, quality bar, latency, and RAM are a **§6 verify item** — it is not yet proven and must not be claimed as shipped until it passes.

**Net:** do not market or reason about CallBrain as "your meetings never leave your Mac." The accurate claim is **"local index and on-device processing; cloud LLM generation by default, with an optional on-device generation mode."** Coordinate any user-facing privacy copy with `regulatory-compliance-sme` / `apple-platform-security-sme`.

## 2. Architecture Verdict

**WINNER: E — Swift-native-first. Confidence HIGH (with the MLX-JIT entitlement caveat in §6).** No Python interpreter in the bundle. **No FastAPI, no HTTP server, no localhost port.** H (hybrid) collapses into E because the hybrid's Python surface shrinks to zero in the shipped app; H survives only as a documented, opt-in, non-bundled fallback toggle (below).

The *only* structural reason pass-1 reached for Python was the AI ecosystem — transcription, diarization, embeddings, vector search. As of 2025–2026 all four have best-in-class **native Swift/CoreML** options on the Neural Engine, and the LLM is a CLI subprocess in every option. The classic gap that forced Python — **pyannote diarization** — is now closed natively by **FluidAudio** (it *is* `community-1` on CoreML). With that gap closed, a bundled Python runtime buys nothing and costs everything the founder forbade.

| Dimension | E: Swift-native-first | B: Python FastAPI sidecar | Winner |
|---|---|---|---|
| Bundle size | Tens of MB; CoreML/MLX models downloaded on first run | Frozen 3.12 + torch + numpy + faster-whisper + pyannote ≈ **multi-GB** `.app` | **E (HIGH)** |
| Cold launch | `.app` to first window `< 1s`; engines lazy-loaded into actors | Spawn interpreter + `import torch` = seconds/launch, or a persistent daemon (extra RAM + lifecycle) | **E (HIGH)** |
| Idle RAM | Tens of MB; engines unload when idle (embedder warm-window — §4) | torch process idle = hundreds of MB resident | **E (HIGH)** |
| Notarization / hardened runtime | Pure Swift + signed CoreML/Metal notarizes clean **IF** MLX-Swift needs no `allow-jit`/`allow-unsigned-executable-memory` (**§6 gate**; CoreML/MLTensor path is the entitlement-free fallback) | PyInstaller+torch is a documented minefield: needs `allow-unsigned-executable-memory` (disables a hardening protection) | **E (HIGH, pending §6 MLX check)** |
| Python-version skew | None in the app (dev/model-prep toolchain still uses Python — §5A) | System python 3.14 too new for ML wheels → bundle + maintain pinned 3.12 forever | **E (HIGH)** |
| Failure modes for a solo non-coder | One process, in-proc actors | Second runtime + localhost port + process supervision + env that rots on OS updates | **E (HIGH)** |
| "Butter" (main-thread discipline) | Actors hop to `@MainActor` only to render; zero IPC serialization | Every query crosses a JSON/HTTP boundary to another process | **E (MED-HIGH)** |
| AI ecosystem access | Native covers transcription/diarization/embeddings/vectors; LLM via subprocess | Marginally broader, but unused here | **B, but irrelevant** |
| Update size (Sparkle delta) | Tens of MB | Re-ship the GB runtime on any dep bump | **E (HIGH)** |

**Smallest-possible Python surface in the shipped app = ZERO by default.** The only sanctioned *runtime* Python is an **opt-in, on-demand, non-bundled diarization helper**, invoked exactly like the LLM and gated behind a per-file "high-accuracy diarization" toggle. It is a `uv run` script (pinned 3.12 in a lazily-created user-space env), takes a WAV, returns diarization JSON, then exits. **Never in the `.app`, never in the hardened runtime, never in the launch/hot path.** It exists solely as insurance if FluidAudio fails the on-device DER gate on real crypto calls — and that must be *proven empirically* before it is ever entertained. (Separately, a **dev/model-prep** Python toolchain exists outside the shipped app — see §5A.)

**FastAPI decision: NO.** Retrieval, ranking, transcription, diarization, and embedding all run **in-process in actor-isolated engines** off the main actor; SwiftUI renders after an explicit `@MainActor` hop. The only out-of-process calls are subprocesses (`claude`/`codex`, optional `ffmpeg`, optional opt-in diarization helper, optional local-LLM helper), each wrapped in a cancellable `Process`/Subprocess actor. Eliminating the web server removes serialization overhead, a port-binding failure mode, and a second supervised runtime.

## 3. Consequences per subsystem (vs pass-1 assumptions)

- **Ingestion.** Transcript-first is unchanged and now *fully native*: Fathom/Fireflies/Otter/Cluely transcripts are parsed/normalized/indexed in Swift (they already carry speaker labels → diarization is skipped). Only raw Google Meet videos lacking a transcript hit the transcription path. **AVFoundation decodes audio first**; ffmpeg is an optional signed CLI helper, not a Python dependency. A `chunk` is the atomic unit: `{chunk_id, meeting_id, speaker, start_word_ts, end_word_ts, normalized_text}`, persisted in SQLite so **re-embedding never requires re-transcription**.
- **Transcription.** Pass-1's faster-whisper/torch path is **deleted**. Default heavy path = **WhisperKit `large-v3-turbo`** (CoreML/ANE, word timestamps, VAD chunking), model downloaded on first run to Application Support. Live/low-power/while-downloading fallback = **Apple SpeechTranscriber** (low app RAM, system-managed asset, also downloaded on first use). Turbo bulk-backlog option = FluidAudio Parakeet TDT v3 (~117× reported, verify). **Offline/first-run: neither path works on a fresh offline machine until its asset downloads — see §1A and §6.**
- **Diarization.** Pass-1's pyannote/torch sidecar is **deleted**. Default = **FluidAudio offline `community-1`** (ANE). Always use the **offline** pipeline (streaming DER is materially worse and does not apply to batch). Word↔speaker join = **interval intersection** (assign each word to the speaker segment containing its midpoint; tie-break to max-overlap). Run diarization and transcription in parallel, then merge. **Benchmark DER (AMI-IHM 17.0% / AMI-SDM 19.9%, official card) is meeting-room, NOT crypto-VoIP-representative; the §6 real-call DER is binding.** Carry the CC-BY-4.0 attribution notice (§5).
- **Retrieval.** One SQLite file is the source of truth: relational metadata + external-content **FTS5** + embeddings as BLOBs. The **usearch HNSW** index is a fast, *rebuildable* derived cache keyed by `chunk_id`. Hybrid query = vector arm (usearch, k=50, predicate pre-filter) + keyword arm (FTS5 BM25, k=50) fused by **RRF (k≈60)** in Swift, then hydrate by `chunk_id`. **Predicate-pre-filtered ANN is a known recall hazard at high filter selectivity (single-meeting / narrow date range) — §6 sub-gate, with over-fetch+post-filter or FTS5/brute-force routing as the documented fallback.** Embeddings = **Qwen3-0.6B via MLX-Swift** behind an `Embedder` protocol with a nomic/ANE fallback. Store an **index manifest** (`model_id, revision, dim, pooling, normalized, prompt_template_version`); version changes write a new versioned table and **atomic-swap after verification — but at scale this is a throttled, resumable background re-index, not an instant swap (§5C).**
- **Packaging.** Pass-1's "SwiftUI front + Python FastAPI back" and the entire Python-packaging chapter (PyInstaller, frozen 3.12, torch/pyannote bundling, `allow-unsigned-executable-memory`, deep-sign of vendored `.so`s) are **deleted**. Replaced by: single pure-Swift app, Developer-ID signed + notarized, **direct-download only**, Sparkle delta updates. **Hardened-runtime entitlements are determined by the §6 MLX-JIT check, not assumed.** **sqlite-vec is static-compiled into the custom SQLite build (never a loadable `.dylib`)** to sidestep library-validation entirely (§5B).
- **Lifecycle / concurrency.** Engines (`TranscriptionActor`, `DiarizationActor`, `EmbeddingActor`, `SearchEngine`, `LLMRunner`) are `actor`s **off `@MainActor`**, created on first use, unloaded when idle — **except the query-embedder, which honors a warm-window keep-alive so Cmd-K stays instant (§4).** `Sendable` value snapshots cross boundaries; `TaskGroup`/`async let` fan-out for multi-section ingest; cancellation on view disappear. Strict Swift 6, **no `@unchecked Sendable`** without a written reason.

## 4. Buttery-Smooth Performance Playbook

**Target hardware (PIN before testing — the frame budget depends on it):** primary = `<pin exact Mac, e.g. M-series MacBook Pro 14" with built-in ProMotion 120Hz display>`; **also test the 60Hz external-monitor case**, because the app will be used on a non-ProMotion external display. The frame budget is **8.3 ms @ 120 Hz (ProMotion) and 16.6 ms @ 60 Hz** — the acceptance gate must state which, and "butter" must hold at **both**.

**Three laws (break only with measurement):** (1) render only what's on screen — body cost never scales with archive size; (2) the main actor only diffs views — all DB/vector/FTS/subprocess/markdown work is on an actor; (3) invalidate at the property, not the object — `@Observable` per-row so a token append or selection touches one view, not the list.

**State & observation.** Use `@Observable` (Observation framework), never `ObservableObject`. One small `@Observable` per surface (`MeetingsListState`, `TranscriptViewerState`, `ChatState`, `SearchState`) — no app-wide god-store. Rows are `Equatable` `Sendable` value types keyed on a stable `id` (never array index, never reused ids). Hoist every O(n) derivation (formatting, attributed-string build, sort) onto the actor; `body` is O(1) and allocation-free.

**Per surface:**
- **(a) Meetings list / timeline.** `List` with constant-height, lightweight rows is fine into low thousands; `Table` for sortable columns but bind selection to `Set<Meeting.ID>` of light ids (documented ~1k-row selection hang). **Page the data, never `SELECT *`** — GRDB date-bucketed pages fetched at a bottom sentinel. Section by month/quarter. Thumbnails decoded off-main + downsampled (`CGImageSourceCreateThumbnailAtIndex`).
- **(b) Transcript viewer (10k+ utterances) — the risk surface.** **Primary (HIGH confidence): AppKit `NSTableView` via `NSViewRepresentable`** — 20 years of cell reuse, the only thing reliably hits the per-refresh frame budget over tens of thousands of variable-height rows on macOS. SwiftUI keeps the chrome; only the scrolling body is AppKit. Click-to-jump = `scrollRowToVisible`. Highlight ranges computed **off-main**, stored as per-utterance `NSAttributedString`, only visible rows re-rendered. **TextKit 2** (`NSTextLayoutManager`; STTextView as reference, MIT). Pre-measure row heights once at import → `heightOfRow` is a dictionary lookup. **Secondary (try-first, must measure): SwiftUI `ScrollViewReader` + `LazyVStack`** with stable ids + pre-measured heights; ship it only if Animation-Hitches stays within the pinned per-refresh budget at 10k rows, else escalate to `NSTableView`. **Never** a single `TextEditor`/`Text` for the whole transcript.
- **(c) Ask-AI chat (streaming subprocess).** `LLMRunner` actor owns the subprocess (**swift-subprocess**, `AsyncSequence` stdout, structured cancellation) and exposes `stream(prompt:) -> AsyncStream<String>`, consumed in a `.task` auto-cancelled on disappear. **Coalesce writes to display cadence (~16–33ms), not token cadence.** Render append-only plain text *while streaming*; parse Markdown (`AttributedString(markdown:)`) **only at closed-block boundaries / on completion**. (Reminder: the prompt this actor sends **egresses retrieved context to the cloud — §1A.**) Avoid WebView renderers; SwiftStreamingMarkdown is a reference design (confirm macOS target).
- **(d) Global Cmd-K search.** `.task(id: query)` so each keystroke cancels the in-flight query; ~150–250ms debounce inside via `Task.sleep` + `isCancelled`. All FTS5 + vector work on the `SearchEngine` actor; results return as `Sendable` rows. FTS results stream first, vector merges via `async let`. **The query embedder must be warm (below) or the first post-idle search stalls on model reload.** Cap top ~20–50, snippet + match range pre-highlighted on the actor.

**Embedder lifecycle vs search latency (reconciled).** "Unload when idle" and "search is instant" conflict for the heavy Qwen3-0.6B-4bit doc-embedder (~300 MB+ resident). Policy:
- **Doc-embedder (heavy):** loaded only during ingest/re-index; unloaded when idle. Not on the search hot path.
- **Query-embedder (light, kept warm):** keep a **small ANE encoder (nomic-embed-text-v1.5)** resident for query embedding, OR keep the heavy embedder warm on a **keep-alive window of N minutes after the last query** (tune N from the §6 cold-vs-warm numbers). The simplest butter-preserving default is **nomic-on-ANE for query embedding always-warm, Qwen3 for corpus embedding**, accepting a documented query/doc encoder asymmetry only if the §6 recall@10 check shows it is acceptable; otherwise keep Qwen3 warm via the window.
- **§6 must record cold-vs-warm first-query latency** so this policy is set by data, not hope.

**Launch & RAM.** No eager graph at `App.init` — build the window first, warm DB/index in a post-first-frame `.task` behind a skeleton. Lazy feature loading (index/embedder/LLMRunner created on first use; unused = zero RAM under unified memory, subject to the query-embedder warm-window above). SQLite opened off-main, WAL, prepared statements. `@SceneStorage` for window/selection/scroll restoration. **Ship no browser engine and no bundled Python runtime** — the largest single launch/RAM win.

**Instruments acceptance thresholds (Release build, pinned target Mac + display, ≥2-year seeded archive):**

| Template | What to record | Threshold |
|---|---|---|
| App Launch | cold/warm, pre-main + main→first-frame | cold `< 1000ms`; first CA frame `< 500ms`; no main-thread I/O before first frame |
| SwiftUI (Xcode 26) | Long View Body Updates; Cause & Effect Graph | **zero** body update > **16ms**; no off-screen/sibling re-render on scroll/stream |
| Hangs / Time Profiler | main-thread during scroll/search/stream | **zero** hangs > **250ms**; no DB/FTS/vector/subprocess/markdown symbol on main |
| Animation Hitches / CA FPS | 10k-row fling; Cmd-K updating | frame time within budget for the panel: **< 8.3ms @120Hz ProMotion**, **< 16.6ms @60Hz external**; **zero** hitches on full-archive scroll on **both** |
| Search latency (cold vs warm) | first query after idle vs warm query; embed time | warm `< 250ms`; cold (post-unload) reload cost recorded and bounded by the warm-window policy (§4) |
| Allocations + Leaks | idle RSS; 30-min scripted session | idle `< 150MB` (aim `< 120MB`) **with the query-embedder warm**; no monotonic growth; **zero** leaked view-state |
| Subprocess streaming (manual) | time-to-first-token; flush cadence | first token `< 150ms`; flush ≤ once/frame; **zero** reflow above the cursor |

**Caveats (verify-on-device):** the macOS 26 SwiftUI Instrument "Cause & Effect Graph" has a reported memory-blowup bug — keep traces short, save before opening. SwiftUI `List`/`Table`/`LazyVStack` macOS scaling figures are community reports, not first-party guarantees — treat the SwiftUI transcript path as unproven until Animation-Hitches passes at 10k rows on the pinned hardware.

### 5A. Dev / model-prep toolchain (Python exists here — NOT in the `.app`)

The "zero Python" claim is precise: **zero Python in the shipped `.app` and in the runtime hot path.** Python remains in the offline supply chain and must be documented for reproducibility:
- **MLX DWQ quantization** — producing/refreshing `Qwen3-Embedding-0.6B-4bit-DWQ` is an MLX (Python, `mlx-lm`) process. The artifact is confirmed to exist (`mlx-community/...-4bit-DWQ`, converted with mlx-lm 0.24.1); **pin the exact repo + commit revision** in the index manifest. If the community artifact is ever unmaintained, the founder must run the Python DWQ pipeline (calibration data + mlx tooling); **document a non-DWQ 4-bit quantization fallback** so the embedding model is never blocked on a single community upload.
- **Any CoreML re-conversion** (`coremltools`, Python) for the swift-embeddings/MLTensor fallback path.
- **Embedding numerical-parity reference** — the §6 "cosine ≥ 0.999 vs reference HF Qwen3" check uses PyTorch/`transformers` (Python) as the reference oracle.
- **Optional diarization insurance helper** — the `uv run` pyannote script (runtime, opt-in, non-bundled).

None of these ship inside the `.app`; all are dev-machine steps. Keep them in a `tools/` makefile with pinned versions.

### 5B. Concrete custom-SQLite build (sqlite-vec + FTS5 + optional SQLCipher in ONE library)

sqlite-vec, FTS5, and optional SQLCipher **cannot be mixed à la carte** — they must be **one** custom SQLite build:
- Use **GRDB's custom-SQLite path** (GRDBCustomSQLite, built via `swiftlyfalling/SQLiteLib`) — or the SQLCipher variant for at-rest encryption. You cannot use system `libsqlite3`/default GRDB and also statically compile sqlite-vec.
- Add the sqlite-vec C amalgamation to the same target; compile with **`-DSQLITE_CORE -DSQLITE_EXTRA_INIT=core_init`** so `sqlite3_vec_init` is registered as a built-in auto-extension (no loadable `.dylib`, no library-validation prompt). Ensure `sqlite3ext.h` is visible to the GRDBCustom target.
- Enable FTS5 in the same build via **`-DSQLITE_ENABLE_FTS5`** (set in `SQLiteLib-USER.xcconfig` / `GRDBCustomSQLite-USER.xcconfig`).
- If at-rest encryption is wanted, build on the **SQLCipher amalgamation** and add sqlite-vec + FTS5 flags into that same single build.
- **§6 asserts** the custom build links and that **FTS5 + sqlite-vec + (optional) SQLCipher coexist in one library** and load under hardened runtime/notarization with no extension prompt.

### 5C. Full-corpus (re-)embed strategy at scale

A manifest version bump (model/revision/pooling/prompt-template change) or a nomic downshift forces a **full re-embed of the entire corpus** into a new versioned table — the "atomic-swap after verification" is the *cutover*, not the *work*. At 1M chunks this is a **multi-hour background batch** and must be budgeted:
- **Order-of-magnitude estimates (to be replaced by §6 measured numbers):** at an assumed ~256 tokens/chunk, a 1M-chunk re-embed is ~256M token-passes. Under a **large-batch (bulk) regime** at the hoped throughput this is on the order of **~1–3 hours**; under a **small-batch (interactive/butter-preserving) regime** it can be **several× longer (many hours)**. These are estimates — the §6 throughput gate at ~1M is what sets the real number.
- **Strategy:** a **throttled, resumable, background re-index** with persisted progress (checkpoint by `chunk_id` range), pausable, low-priority QoS so it never fights the foreground; the old versioned table stays serving search until the new one passes the parity/recall check, then atomic-swap. Surface a progress UI ("re-indexing N% — search uses the previous index meanwhile").
- **Tension reconciled:** ingest/re-index may use the **large-batch doc-embedder** (throughput) precisely because it runs in the background at low QoS and the heavy embedder is unloaded when idle; the **foreground query path** uses the **warm light query-embedder** (butter). The two regimes are separated by lifecycle, not in conflict.
